Relatório Técnico de Reorganização

Projeto: INOVATECH ASTRO VERDE

Data: 23/04/2026

Objetivo: organizar, padronizar e preparar o software para integração futura com ESP32 + sensor de pH + bomba, mantendo o funcionamento existente.

1. Resumo Executivo

A base foi reorganizada sem recriar do zero, com foco em separação de responsabilidades, legibilidade e continuidade do projeto.

Foram adicionados estado central do sistema, rotas dedicadas para integração física, padronização de respostas HTTP no backend e centralização de chamadas HTTP no frontend.

2. Diagnóstico da Estrutura Original

  • Frontend e backend já estavam separados, porém com alguns acoplamentos de fluxo e padronização incompleta.
  • Respostas da API eram heterogêneas entre diferentes controladores.
  • Lógica de controle da bomba por pH não estava isolada em serviço próprio.
  • Faltava endpoint de troca de modo (simulado / real).
  • No frontend, endpoints não estavam centralizados em constantes e faltava cliente HTTP comum.
  • Estado de sistema voltado à integração física (modo, origem de leitura, última atualização) não estava explícito.
  • Módulo de automação tinha risco de duplicar listeners em reentrada de aba.

3. Reorganização Aplicada

Backend

  • server/src/routes: rotas HTTP
  • server/src/controllers: camada HTTP
  • server/src/services: regras de negócio
  • server/src/state: estado central em memória
  • server/src/utils: utilitários compartilhados
  • server/src/config: configurações

Frontend

  • assets/js/modules: módulos de interface
  • assets/js/services: integração com API
  • assets/js/constants: endpoints centralizados
  • assets/js/utils: utilitário HTTP
  • assets/js/state.js: estado global da UI

4. Preparação para Camada Física

Estado central do sistema

  • phAtual
  • bombaLigada
  • modoAtual (simulado ou real)
  • ultimaAtualizacao
  • origemLeitura (simulada ou hardware)

Endpoints principais garantidos

  • GET /api/health
  • GET /api/state
  • POST /api/sensor/ph
  • POST /api/control/pump
  • POST /api/mode

Regra de automação da bomba

  • Se pH > 7 → bomba ligada
  • Se pH <= 7 → bomba desligada

A regra foi isolada em serviço próprio para evitar acoplamento na rota/controller.

5. Padronização de Respostas da API

Sucesso

{ "status": "success", "message": "...", "data": {} }

Erro

{ "status": "error", "message": "..." }

Padronização aplicada via utilitário compartilhado server/src/utils/httpResponse.js.

  • Fluxo de backend com controllers/services/repositories já utilizado.

9. Validação Técnica Executada

  • Verificação de sintaxe (node --check) nos arquivos críticos.
  • Smoke tests de API em ambiente local com sucesso.
  • Validação das rotas novas e das rotas legadas após padronização.

10. Como Rodar

  1. No backend: cd server e npm start.
  2. No frontend: servir a raiz do projeto em servidor estático (ex.: Live Server).
  3. Definir fonte de dados em assets/js/state.js:
    • dataSource: "mock" para simulação local.
    • dataSource: "api" para backend Node.js.
